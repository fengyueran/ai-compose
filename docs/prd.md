# Prompt Workbench V1 产品需求文档

## 1. 文档说明

本文档用于定义 Prompt Workbench V1 的产品目标、用户问题、范围边界、核心流程与成功标准。

本文档是产品需求文档，不是技术设计文档。重点回答以下问题：

- 这个产品是什么
- 这个产品为谁服务
- 这个产品解决什么问题
- 第一版要做到什么程度

## 2. 产品定义

Prompt Workbench 是一个面向开发者的提示词工作台，用于按语义分类管理提示词片段，并通过桌面端能力将组合结果直接应用到用户级 AI 编辑器配置目录或规则文件。

这不是聊天工具，不提供消息流、对话输入框或模型回复区。它的核心价值不在于“和 AI 对话”，而在于“组织、选择、组合、维护并落地用户级 AI 指令资产”。

V1 的推荐产品形态为 `React + Tauri` 桌面应用，而不是纯浏览器前端。首版就需要稳定访问和写入用户级 AI 编辑器配置目录，浏览器文件选择与授权模型不适合作为长期主路径。

V1 支持多个用户级编辑器目标，优先覆盖个人开发者高频使用的 AI 编码工具。

虽然 V1 当前只支持提示词片段能力，但 UI 架构必须从第一版开始考虑未来扩展，以便后续加入更多编辑器目标与更多配置项，例如 `MCP`、`Skills`、`Profiles` 等，而不需要推翻整体信息架构。

## 3. 目标用户

V1 的目标用户是需要维护个人 AI 编码环境与提示词资产的开发者，尤其包括以下人群：

- 经常在项目中使用多个 AI 编码工具辅助开发的个人开发者
- 需要为自己的 AI 编辑器使用环境维护稳定工作规范的开发者
- 想把零散提示词沉淀为结构化个人配置资产的高级用户

这类用户的共同特征是：

- 关注用户级提示词配置的长期维护，而不是临时聊天
- 需要把提示词沉淀到稳定的用户级配置位置，而不是散落在本地笔记里
- 希望降低从零撰写项目提示词的成本

## 4. 背景与用户问题

当前用户级 AI 提示词维护存在以下典型问题：

### 4.1 提示词内容分散

很多提示词散落在：

- 临时对话中
- 本地笔记中
- 不同项目的说明文件中
- 团队成员各自维护的私人模板中

这导致提示词难以沉淀、难以复用，也很难形成稳定的个人 AI 配置规范。

### 4.2 缺少稳定的语义组织方式

用户级 AI 提示词通常包含多种不同语义内容，例如：

- 核心原则
- 安全约束
- 代码质量要求
- 验证要求
- 输出格式要求

如果这些内容都堆在一整段长提示词里，会出现以下问题：

- 难以定位和修改局部规则
- 难以按场景重组
- 难以逐步沉淀预设

### 4.3 修改后难以稳定落地

即使用户已经有一套个人提示词配置，也常常面临这些问题：

- 不知道不同编辑器应该写到哪个用户级目录或文件
- 不敢直接修改已有编辑器配置
- 改动容易覆盖掉其他已有内容
- 缺少可控的受管更新方式

如果仅依赖浏览器前端，还会额外出现这些体验问题：

- 需要用户手动选择目标文件
- 难以稳定定位 `~/.codex`、`~/.cursor`、`~/.gemini` 等用户级目录
- 后续扩展到更多编辑器时交互会越来越像文件工具，而不是配置工作台

因此，用户需要的不是“复制一段 prompt”，而是“稳定地把提示词应用到自己的 AI 编辑器使用环境中”。

## 5. 产品目标

Prompt Workbench V1 的目标是帮助用户完成以下闭环：

1. 浏览一套面向 AI 编码代理场景的官方预设提示词片段
2. 按语义分类选择需要的片段
3. 编辑自己的片段内容
4. 将所选片段组合成最终用户级提示词
5. 通过桌面端能力直接应用到用户级 AI 编辑器配置目录或规则文件

第一版的重点不是覆盖所有 AI 工具，也不是实现复杂编排能力，而是先把“用户级提示词维护与多编辑器落地”这条路径做完整、做稳定。

## 6. V1 核心能力

### 6.1 内置官方预设片段库

V1 必须内置一套默认可用的提示词片段库，服务 AI 编程代理场景。

这套预设片段应优先参考现有 `Codex / ECC` 指令体系中真正稳定、可复用、适合作为行为约束的内容，并用中文提供，同时保持对不同 AI 编辑器的通用性。

预设的主要作用包括：

- 降低用户从空白开始编写提示词的成本
- 提供推荐的语义拆分方式
- 形成产品的默认最佳实践

预设内容应遵循以下筛选原则：

- 优先保留“行为规范”和“工作方式”类内容
- 优先保留可跨项目复用的稳定原则
- 优先保留能直接影响编码、验证、审查质量的约束
- 删除偏说明文档、偏工具介绍、偏实现细节的内容
- 删除不适合作为长期提示词片段的临时性信息

结合现有 Codex/ECC 补充内容，以下类型适合作为预设片段来源：

- 编码工作原则
- 计划与执行方式
- 测试与验证要求
- 安全要求
- 代码审查要求
- Git 使用约束
- 输出风格与协作方式

以下类型不适合作为提示词片段，应从预设中删除或避免纳入：

- 模型推荐表
- MCP 背景说明文字
- 具体脚本实现说明
- 配置文件内部结构细节
- 一次性迁移说明
- 过度依赖当前仓库结构的描述

### 6.2 按语义分类浏览和选择片段

提示词片段不是按“单条完整 prompt”管理，而是按语义分类组织。用户应能够从分类中查看、筛选和选择片段。

首版优先支持以下语义分类：

- 核心原则
- 计划与执行
- 测试与验证
- 安全约束
- 代码审查
- Git 工作流
- 输出与协作风格

这些分类应尽量贴近真实 Codex/ECC 使用场景，而不是泛化成过宽的提示词目录。

例如，基于你提供的 Codex/ECC 补充内容，第一版更适合作为预设片段的方向包括：

- “先计划再执行”
- “修改前后必须验证”
- “不要硬编码敏感信息”
- “优先识别高风险问题并给出可验证结论”
- “使用仓库既有技能与工作流表面”
- “遵循约定的 Git 与审查方式”

相对地，以下内容不应直接作为首版预设片段主体：

- “Routine coding 使用哪个模型”
- “MCP 服务器列表说明”
- “Context7 canonical section name”
- “features.multi_agent 的开启方法”

这些内容更适合作为帮助文档、设置说明或后续配置域资料，而不是 Prompt 片段本身。

### 6.3 编辑用户片段

除了官方预设，用户还应能够维护自己的片段内容。

V1 中，用户编辑的重点是“文本内容维护”，而不是复杂结构化编排。交互应尽量轻量，避免让第一版变成规则引擎。

### 6.4 组合最终提示词

用户选择多个语义片段后，系统应生成一份最终用户级提示词预览。

这里的目标不是做聊天输入，而是为后续通过桌面端能力应用到用户级编辑器目标文件提供清晰、稳定的产物。

### 6.5 应用到用户级编辑器配置目录或规则文件

V1 的最终动作不是复制文本，而是通过桌面端宿主能力将组合结果直接写入用户级编辑器配置目录中的受管目标文件或规则文件。

当前 V1 已支持的用户级目标包括：

- `Codex`：`~/.codex/AGENTS.md`
- `Cursor`：`~/.cursor/AGENTS.md`
- `Antigravity`：`~/.gemini/GEMINI.md`

应用策略采用受管区块模式：

- 如果目标文件中已有受管区块，则仅更新区块内容
- 如果目标文件存在但没有受管区块，则追加新区块
- 如果目标文件不存在，则创建新的受管目标文件

这一能力是 V1 的关键产品价值，因为它把“提示词编辑”真正闭环到了“用户级环境落地”。

为了保证这一能力稳定可用，V1 应采用桌面端原生文件访问，而不是把浏览器文件选择器作为正式产品路径。

## 7. 非目标

以下内容明确不属于 V1 范围：

- 聊天界面
- 模型调用
- AI 回复展示
- `Claude Code` 支持
- 多编辑器一键同步写入
- 云同步
- 团队协作
- 复杂版本管理
- 条件片段规则
- 模板变量系统
- MCP、skills、profile 联动

以下内容也不作为 V1 正式方案：

- 纯浏览器文件选择器驱动的长期写入方案
- 依赖用户手动挑选目标文件的正式交互路径

V1 的目标是清晰和稳定，不追求大而全。

## 8. 核心产品决策

### 8.1 管理对象是提示词片段

V1 的最小管理单元是“提示词片段”，不是完整聊天 prompt，也不是完整对话模板。

这样设计的原因是：

- 更适合按语义拆分
- 更适合后续重组
- 更适合沉淀官方预设与用户自定义内容

### 8.2 片段按语义分类组织

V1 不采用无结构的大列表，也不采用复杂多层模块树。首版采用“语义分类 + 片段列表”的组织方式。

### 8.3 必须内置官方预设片段

如果没有预设，第一版会退化成一个普通文本编辑器。预设片段库是第一版的重要产品价值，不作为可选功能。

预设片段库的内容质量标准应明确为：

- 以中文为主
- 优先保留高复用、高约束、高信号内容
- 删除偏说明性、偏罗列性、偏配置百科式内容
- 保证每个片段都能直接提升目标 AI 编辑器的行为质量，而不是仅提供背景知识

### 8.4 最终动作是应用，不是复制

产品的终点不是“生成一段文本让用户手动复制”，而是“通过桌面端能力将结果应用到用户级编辑器配置目录或规则文件”。

这意味着产品要对“提示词如何落地到用户使用环境”负责。

### 8.5 应用策略为受管区块更新

V1 采用“无区块则追加，有区块则更新”的策略。

该策略的目的包括：

- 保留用户已有的非受管内容
- 降低误覆盖风险
- 让系统的改动边界清晰可控

### 8.6 UI 必须为多编辑器与多配置域扩展预留结构

虽然第一版当前最成熟的是 `Prompt` 工作流，但 UI 不能设计成一次性页面或单用途表单。

第一版 UI 需要满足以下扩展前提：

- 当前已支持 `Codex`、`Cursor`、`Antigravity` 三个用户级目标
- 后续可以继续加入新的目标编辑器，例如 `Claude Code`
- 后续可以加入新的配置域，例如 `MCP`、`Skills`、`Profiles`
- 新目标和新配置域加入后，主导航、工作区结构和预览区不需要整体重做

这意味着第一版就应采用“工作台式信息架构”，而不是围绕单一操作路径硬编码页面结构。

## 9. 核心用户流程

V1 的核心用户流程如下：

1. 用户打开 Prompt Workbench
2. 选择目标编辑器
3. 浏览官方预设片段分类
4. 选择需要纳入提示词的片段
5. 按需编辑用户片段
6. 在界面中预览最终组合结果
7. 通过桌面端能力将结果应用到目标编辑器的用户级配置目录或规则文件

该流程应满足两个要求：

- 用户无需从零手写整份用户级提示词
- 用户无需手动复制粘贴到目标文件

## 10. 成功标准

如果 V1 成功，应该能够满足以下结果：

### 10.1 用户层成功标准

- 用户可以从预设片段起步，而不是空白开始
- 用户可以按语义方式理解和组织个人提示词配置
- 用户可以在不手动改文件的情况下完成用户级提示词落地

### 10.2 产品层成功标准

- 第一版能够独立完成多个用户级编辑器目标的提示词维护闭环
- 组合结果能够稳定写入各用户级目标文件的受管区块或规则文件
- 受管写入不会破坏文件中非受管内容

### 10.3 评审层成功标准

阅读本文档后，评审者应能无歧义回答以下问题：

- 这是不是聊天产品
- 第一版服务谁
- 第一版支持哪些目标工具
- 为什么必须有预设片段
- 最终产物如何落地
- 第一版明确不做什么

如果读完 PRD 后，实现者仍需自行决定以下问题，则说明 PRD 不完整：

- 是复制还是直接应用
- 管理完整 prompt 还是片段
- 第一版支持哪些目标工具

## 11. V1 页面级预期

虽然本文档不展开技术实现，但 V1 的产品形态应满足以下页面级预期：

- 存在一个非聊天的工作台主界面
- 界面结构应明显区分“目标编辑器”“配置域”“当前工作内容”“最终应用结果”
- 能看到分类与片段列表
- 能看到当前片段编辑区域
- 能看到最终组合结果预览
- 能触发应用到当前选中用户级目标文件的动作

这些页面级预期用于约束产品形态，不等同于详细交互规格。

从信息架构角度，第一版 UI 应具备以下扩展意识：

- 左侧导航层应可容纳未来的编辑器切换与配置域切换
- 中间工作区应服务于当前选中的配置域内容编辑，而不是只服务提示词
- 右侧预览区应可从“最终 Prompt 预览”演进为“目标工具适配结果预览”
- 操作区应可从“应用到当前编辑器”扩展为“应用到不同编辑器目标”

在视觉和交互方向上，第一版应参考工作台型产品，而不是聊天产品或普通表单页。界面应强调：

- 清晰的信息层级
- 高密度但不拥挤的配置体验
- 可扩展的导航结构
- 明确的预览与应用反馈

UI 质量上应遵循以下原则：

- 保持非聊天形态，不引入消息流和对话气质
- 保证键盘可达、焦点清晰、对比度达标
- 所有主要交互状态都有明确反馈
- 设计风格保持一致，不因未来扩展而破坏整体工作台结构

## 12. 风险与约束

V1 需要特别关注以下产品风险：

### 12.1 范围膨胀风险

如果过早引入多工具适配、变量系统、skills 联动或聊天能力，第一版会迅速失焦。

### 12.2 误写项目文件风险

如果应用策略不清晰，可能误覆盖用户已有编辑器配置内容。因此产品必须坚持受管区块边界。

### 12.3 预设质量风险

第一版预设片段数量不必多，但质量必须高。若预设质量不足，产品将失去“开箱即用”的价值。

## 13. 后续规格待定义

在本 PRD 确认后，下一阶段需要补充以下规格文档，但不在本文展开：

- UI 信息架构与导航扩展方案
- 信息架构与页面交互规格
- 片段数据模型与文件格式
- 官方预设片段目录与内容规范
- 组合顺序与排序规则
- 不同用户级目标文件的受管区块格式与写入规则
- 应用前后的校验与错误提示策略
- Tauri 宿主命令边界与前后端职责划分

## 14. 结论

Prompt Workbench V1 的本质，是一个围绕用户级 AI 编辑器提示词维护场景构建的非聊天工作台。

它通过“官方预设片段 + 语义分类组织 + 桌面端直连用户级目标文件”这条路径，帮助开发者把零散提示词沉淀为稳定、可维护、可落地的用户级 AI 指令资产。
