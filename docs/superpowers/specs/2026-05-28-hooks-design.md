# Hooks 配置支持设计

日期：2026-05-28
分支：`feat/configure-hooks`
状态：已确认，待用户审阅

## 背景

`ai-compose` 目前已经支持三类受管配置：

- Prompt
- MCP
- Skills

这三类能力都遵循相似模式：

- 前端工作台维护统一状态
- Tauri 命令负责读取和写回目标编辑器配置
- 只改写 AI-COMPOSE 自己的受管内容
- 非受管内容保持不变

本次新增第四个域：`Hooks`。目标是在 `Codex`、`Cursor`、`Antigravity` 三类 AI 工具配置中提供 hooks 的统一管理入口。

## 目标

- 在工作台中新增 `Hooks` 域，与 `Prompt / MCP / Skills` 并列
- 首版支持 `Codex + Cursor + Antigravity`
- 首版以 `plain/raw` 编辑为主，不做结构化表单编辑
- 首版允许统一维护 hooks 元信息，再按编辑器分别启停和写回
- 首版读取已有 hooks 供参考，但新增和修改只作用于 AI-COMPOSE 受管部分
- 为后续结构化 hooks 编辑保留数据模型和扩展边界

## 非目标

- 首版不提供结构化 hooks 表单编辑器
- 首版不尝试自动改写用户原有的非受管 hooks
- 首版不做“部分成功自动降级”策略
- 首版不引入开放式变量模板系统
- 首版不实现新的执行引擎，只负责配置内容的管理、校验和写回

## 已确认决策

- 支持目标编辑器：`Codex`、`Cursor`、`Antigravity`
- 编辑方式：首版直接使用 `plain/raw` 编辑
- 长期方向：后续可在 plain 之上补结构化编辑
- 统一管理方式：工作台提供统一 hooks 管理入口，但每个编辑器可独立开关
- 预留触发点：`执行前`、`执行后`、`失败后`、`提交前`
- 失败策略：每条 hook 可配置 `阻断后续` 或 `仅告警`
- 内置变量：首版只支持小集合，例如 `changed_files`、`project_root`、`branch`
- 不支持的编辑器能力：切换到对应编辑器或尝试启用时直接报错
- 现有 hooks 处理：读取展示，但只读参考；AI-COMPOSE 仅改写自己的受管部分
- 命令复杂度：一条 hook 可对应单条命令，也可对应顺序命令组

## 用户体验

### 工作台层级

在现有顶层域选择中新增 `Hooks`，与 `Prompt / MCP / Skills` 并列。

`Hooks` 域首版提供以下能力：

- 选择当前查看的编辑器
- 查看该编辑器配置文件中的 hooks 相关状态
- 查看现有非受管 hooks 的只读摘要或只读原文
- 查看和编辑 AI-COMPOSE 受管 hooks 原文
- 对当前编辑器启用或禁用 AI-COMPOSE 受管 hooks
- 配置 hook 的统一元信息
- 触发“应用到当前编辑器”

### 首版编辑体验

首版不做复杂交互式表单，而采用“元信息 + 原文编辑区”的组合：

- 元信息由工作台维护，包括名称、触发点、失败策略、编辑器开关
- 具体 hooks 内容由用户直接编辑 plain/raw 文本
- 编辑器之间可以共享一份逻辑上的 hook 条目，但每个编辑器的 raw 内容单独保存

这样可以先解决三端配置格式差异较大的问题，同时为后续结构化编辑保留扩展点。

## 设计概览

### 核心原则

- 受管边界清晰：只修改 AI-COMPOSE 创建的内容
- 显式失败：不兼容或配置非法时直接报错
- 统一入口：用户在一个 Hooks 域中管理三端配置
- raw 优先：先把链路跑通，再考虑更高阶抽象
- 可演进：当前数据模型不阻塞未来结构化能力

### 总体架构

实现沿用现有三层结构：

1. 前端状态层
2. 前端 hooks 面板
3. Tauri 读写与校验层

前端负责组织统一的 hooks 视图状态和编辑状态，Tauri 负责：

- 定位不同编辑器的目标配置文件
- 读取当前文件内容
- 提取现有 hooks 信息
- 提取或替换 AI-COMPOSE 受管 hooks 区块
- 校验 raw 内容与编辑器支持性
- 将变更写回文件

## 数据模型

### 统一元信息模型

虽然首版直接编辑 raw 内容，前端仍需维护统一元信息，建议最少包含：

- `id`：hook 条目稳定标识
- `name`：hook 展示名称
- `trigger`：统一触发点枚举
- `failurePolicy`：`block` 或 `warn`
- `enabledEditors`：按编辑器维度记录是否启用
- `rawByEditor`：每个编辑器对应的 raw 内容
- `variablesUsed`：从 raw 内容中解析出的内置变量集合
- `mode`：固定为 `raw`，为后续结构化模式保留字段

### 编辑器状态模型

每个编辑器的 hooks 状态建议包含：

- `enabled`：该编辑器上的 AI-COMPOSE hooks 是否启用
- `targetPath`：目标配置路径
- `managedHooksRaw`：当前受管 hooks 原文
- `detectedUserHooksRaw`：检测到的非受管 hooks 只读原文或摘要
- `validationErrors`：当前编辑器的配置错误信息
- `supportedTriggers`：该编辑器当前支持的触发点集合

### 说明

首版不要求三端共用完全相同的 raw 文本。统一的是工作台入口、元信息与应用流程，不是强行用一份文本覆盖三端。

这能降低错误翻译风险，也更符合当前“plain 先落地”的范围。

## 受管边界

### 读策略

加载编辑器 hooks 状态时：

1. 读取完整配置文件
2. 识别现有用户 hooks 内容
3. 识别 AI-COMPOSE 受管 hooks 内容
4. 将用户 hooks 作为只读参考展示
5. 将 AI-COMPOSE 受管内容作为可编辑内容进入工作台状态

### 写策略

应用到编辑器时：

1. 重新读取目标配置文件
2. 只替换 AI-COMPOSE 受管 hooks 区块
3. 保持非受管 hooks 内容不变
4. 将更新后的完整配置写回原文件

### 受管区块规则

Prompt 当前已经使用 `BEGIN/END AI-COMPOSE` managed block。Hooks 首版沿用相同思路，但受管内容应具有明确语义标签，避免与 Prompt、MCP、Skills 的受管内容冲突。

建议使用独立的 hooks 受管标记，或在现有 AI-COMPOSE 受管范围中采用可区分的 hooks 子区块。最终实现时应选一个能满足以下约束的方案：

- 能稳定定位 hooks 受管内容
- 不会误删其他受管域内容
- 能重复应用而不发生区块漂移
- 能在文件为空时安全初始化

## 兼容性与报错规则

### 编辑器开关

每条 hook 在每个编辑器上都可单独启用或禁用。

### 不支持场景

如果某个编辑器不支持：

- 某个触发点
- 某种 raw 格式
- 某个变量
- 某个必需字段

则行为如下：

- 用户切换到该编辑器时，界面展示明确错误原因
- 用户尝试在该编辑器启用对应 hook 时，操作失败
- 用户尝试应用到该编辑器时，Tauri 命令返回错误

系统不做静默跳过，也不做自动降级。

### 失败策略

每条 hook 的失败策略是用户配置的一部分：

- `block`：失败应阻断后续流程
- `warn`：失败只告警，不中断主流程

首版先管理该配置值本身，不实现新的执行器逻辑。

## 变量支持

首版只支持小范围内置变量，目的是覆盖最常见的命令拼接需求，同时控制实现复杂度。

建议首批变量：

- `changed_files`
- `project_root`
- `branch`

### 变量规则

- 变量名必须来自受支持白名单
- 非白名单变量在保存或应用时直接报错
- 变量解析只做轻量校验，不引入脚本式表达能力
- 变量占位格式在三端应尽量保持一致，避免首版就引入多套模板语法

## 实现方案

### 前端

建议新增：

- `Hooks` 域枚举和值切换逻辑
- hooks 专属 store 状态与 actions
- hooks 面板组件
- hooks API 调用封装

前端职责：

- 组织 hooks 的编辑状态
- 展示当前编辑器的受管和非受管内容
- 处理开关、错误提示和应用反馈
- 在本地做轻量校验
- 调用 Tauri 命令完成真实读写

### Tauri

建议新增与现有 prompt/mcp/skills 平行的 hooks 模块，至少包含两类命令：

- `load_editor_hooks_states`
- `apply_hooks_to_editor_target`

Tauri 职责：

- 解析各编辑器目标路径
- 读取配置文件
- 提取 hooks 相关区块
- 校验 raw 内容是否合法
- 校验触发点、变量与编辑器能力是否匹配
- 替换受管 hooks 区块并写回

### 配置格式处理

由于三端配置格式可能不同，首版不做“统一 raw 自动翻译”。首版只保证：

- 在同一个 Hooks 域中统一管理三端配置
- 元信息结构一致
- 应用流程一致
- 每个编辑器保存自己的 raw 内容

后续如果需要“一份结构化配置翻译三端”，应在结构化编辑版本中单独设计。

## 数据流

### 加载流程

1. 工作台进入 `Hooks` 域
2. 前端调用 `load_editor_hooks_states`
3. Tauri 读取三端目标文件
4. Tauri 返回每个编辑器的状态、路径、受管内容、非受管摘要、错误信息
5. 前端将结果注入 hooks store
6. 用户选择编辑器并查看或编辑对应 raw 内容

### 应用流程

1. 用户在当前编辑器下编辑 raw 内容
2. 用户点击应用
3. 前端做基础字段校验
4. 前端调用 `apply_hooks_to_editor_target`
5. Tauri 再次读取原文件
6. Tauri 校验编辑器能力、变量与 raw 内容
7. Tauri 只替换 hooks 受管部分
8. Tauri 写回并返回结果
9. 前端展示成功或失败反馈

## 错误处理

必须覆盖以下错误场景：

- 目标配置路径不存在但目录可创建
- 目标配置路径不可解析
- 配置文件读取失败
- 配置文件写回失败
- 现有文件格式无法安全识别 hooks 受管区块
- raw 内容为空但当前编辑器已启用
- 使用了不支持的变量
- 使用了当前编辑器不支持的触发点
- 尝试启用当前编辑器不支持的 hook

错误提示应满足两个要求：

- 对用户可诊断，能指出是哪个编辑器、哪条 hook、哪类错误
- 不泄露不必要的内部细节

## 测试策略

### 单元测试

- 受管 hooks 区块插入
- 受管 hooks 区块替换
- 受管 hooks 区块移除
- 非受管内容保持不变
- 变量白名单校验
- 编辑器支持性校验
- 触发点与失败策略的序列化行为

### 集成测试

- 三端目标文件加载
- 空文件初始化写回
- 含已有非受管 hooks 文件的写回
- 仅 hooks 受管区块被修改
- 某编辑器不支持时返回明确错误

### UI 测试

- 域切换到 Hooks
- 编辑器切换
- raw 内容编辑
- 应用成功反馈
- 应用失败反馈
- 非受管 hooks 只读展示

## 迁移与演进

该方案是“raw 优先”的首版方案，后续演进方向如下：

1. 在不破坏现有受管格式的前提下引入结构化 hooks 模型
2. 允许常见场景使用结构化编辑器生成 raw 内容
3. 对高级配置继续保留 raw 逃生口

这意味着首版实现时应避免把 hooks store、hooks API 和 hooks 文件格式写死成“只能 raw 字符串”的单一形态。即便 UI 首版只有 raw 编辑器，内部状态也应为未来结构化模式留出字段。

## 风险与取舍

### 选择 raw 优先的收益

- 更快落地
- 更符合三端配置格式存在差异的现实
- 降低错误翻译和 silent corruption 风险

### 选择 raw 优先的代价

- 首版统一维护体验弱于结构化方案
- 三端之间不能真正“一份文本自动翻译”
- 用户仍需理解目标编辑器 hooks 语法

### 本次取舍

在当前范围下，优先保证：

- 安全地读写
- 明确的受管边界
- 清晰的错误反馈
- 不破坏现有用户配置

而不是追求首版就提供完整的结构化抽象。

## 成功标准

- 工作台出现 `Hooks` 域，且可进入使用
- 三端 hooks 状态可加载
- 用户可查看已有非受管 hooks 的只读信息
- 用户可编辑并应用 AI-COMPOSE 受管 hooks raw 内容
- 应用后仅 hooks 受管部分被修改
- 不支持的编辑器能力会直接报错
- 相关测试覆盖受管写回、错误处理和基础 UI 流程
